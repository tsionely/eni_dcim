# 03 — ראייה

המצלמה היא חיישן המיקום היחיד שלנו. מסמך זה מפרט את הצינור מ-datagram של UDP ועד pose יחסית-לשער, את גלאי סבב 1, את תוכנית הכיול, ואת הגיאומטריה של החלון העיוור.

## 1. צינור המצלמה — מקצה לקצה

```
UDP 5600 ──► vision_rx.py ──► LatestValue[Frame] ──► Perception thread ──► LatestValue[Detection]
 (chunks)     (הרכבה, GC)      (JPEG + sim_time_ns)    (decode→detect→PnP)     (gate_rel pose)
                                                                                      │
                                                                              state_estimator.py
                                                                              (תיקון VIO-lite)
```

### 1.1 קליטה והרכבה — `src/aigp/io/vision_rx.py`

פריים JPEG מפוצל לפקטות UDP. כותרת כל פקטה — struct `"<IHHIIQ"`:

```
frame_id       uint32   מזהה פריים
chunk_id       uint16   אינדקס chunk בתוך הפריים
total_chunks   uint16   כמה chunks בפריים
jpeg_size      uint32   גודל ה-JPEG השלם
payload_size   uint32   גודל ה-payload בפקטה זו
sim_time_ns    uint64   חותמת זמן סימולטור
```

כללי ההרכבה:

- טבלת פריימים-בבנייה ממופתחת ב-`frame_id`; פריים שהושלם (כל `total_chunks` הגיעו) נדחף כ-`LatestValue` — כולל `sim_time_ns` שמומר לציר הזמן האחוד דרך `SimClock`.
- **‏partial-frame GC**: פריים שלא הושלם עד שהגיע פריים חדש יותר — נזרק ונמנה. UDP מאבד פקטות; פריים חסר-chunk הוא מקרה שגרתי, לא חריג. בלי ה-GC הזה ה-template המקורי דלף זיכרון.
- אין טיפול ב-reorder מעבר לזה: אם chunks מגיעים בסדר שרירותי בתוך פריים — נתמך (מיקום לפי `chunk_id`); פריים ישן שמגיע אחרי חדש — נזרק.
- ‏decode של ה-JPEG **לא** קורה כאן — רק ב-Perception thread. ה-RX thread נשאר זול.

### 1.2 עיבוד — Perception thread

רץ בקצב הגעת הפריימים. לכל פריים: decode (‏cv2, משחרר GIL) ← ‏`GateDetector.detect()` ← אם נמצא שער: solvePnP ← פרסום `Detection` (פינות בפיקסלים, gate_rel pose, חותמת זמן, ציוני איכות) ל-`LatestValue`. אם הפריימים מגיעים מהר מקצב העיבוד — מעבדים תמיד את האחרון (סמנטיקת `LatestValue`), לא צוברים תור.

## 2. גלאי סבב 1 — `gate_detector_hsv.py`

סביבת סבב 1 היא desaturated עם שערים בניגודיות גבוהה — בדיוק המקרה שגלאי קלאסי מצטיין בו. האלגוריתם, צעד-צעד:

1. **המרה ל-HSV** וסף (threshold) על טווח הצבע/הבהירות של השער. גבולות הסף — רשומות `ParamSet` (ולכן ניתנות לכוונון אוטומטי אם הגלאי מפספס).
2. **ניקוי מורפולוגי** קל (open/close) להסרת רעש נקודתי.
3. **חילוץ contours** (‏`cv2.findContours`) וסינון לפי שטח מינימלי ויחס צלעות סביר.
4. **התאמת מרובע קמור (convex quad fit)**: קירוב פוליגון (`approxPolyDP`) וחיפוש ארבע פינות; מסגרת שער נראית כטבעת — לוקחים את הקונטור הפנימי/החיצוני באופן עקבי (בחירה קבועה, מתועדת מול מידות השער הנומינליות).
5. **סידור פינות (corner ordering)**: מיון דטרמיניסטי ל-top-left, top-right, bottom-right, bottom-left — תנאי הכרחי להתאמה נכונה לנקודות המודל ב-PnP.
6. **‏`cv2.solvePnP`**: ארבע פינות התמונה מול ארבע פינות מודל השער במידותיו הנומינליות ← ‏rotation + translation של השער במערכת המצלמה ← היפוך ל-pose של הרחפן יחסית לשער.
7. **בדיקות שפיות**: מרחק בטווח הגיוני, reprojection error קטן מסף, השער "לפנים" — כישלון בכל אחת פוסל את ה-detection (עדיף להחמיץ פריים מלהזין pose שגויה ל-estimator).

בחירת שער כשנראים כמה: הגדול בתמונה (הקרוב), בכפוף לקונסיסטנטיות עם ה-gate_rel הנוכחית. ה-planner ממילא מאשר התקדמות רק דרך `active_gate_index`.

## 3. הנחת מידות השער

- ‏PnP דורש את המידות הפיזיות של האובייקט. מיקומי השערים מאופסים בטלמטריה, אבל **המידות** של שער תקני הן נתון סביר להנחה.
- המידות הנומינליות (רוחב/גובה פתח) הן רשומת `ParamSet` — לא קבוע בקוד.
- אם ההנחה שגויה: השגיאה היא **סקאלה** על המרחק המשוערך (שער "גדול מהאמור" ייראה קרוב מהאמיתי). הכיוונים נשארים נכונים; פרופיל המהירות ונקודת ה-commit מוטים.
- מיתיגציה: הפרמטר ניתן לכוונון בקמפיין (טעות סקאלה עקבית תתגלה כ-commit מוקדם/מאוחר שיטתי), והוא ברשימת הסיכונים במסמך [05](05-avney-derech.md).

## 4. מודל המצלמה וכיול ה-FOV — `camera.py`

- מודל pinhole פשוט: ‏focal length בפיקסלים נגזר מ-FOV אופקי ומרוחב התמונה; מרכז אופטי מונח במרכז התמונה; עיוות עדשה מונח זניח עד שיוכח אחרת.
- **ה-intrinsics אינם מפורסמים** על ידי הסימולטור — לכן ה-FOV חי ב-`ParamSet` ומכויל אמפירית.
- תוכנית הכיול (על מכונת ה-Windows, במסגרת Phase 3):
  1. הצבת הרחפן ב-hover מול שער במרחק שמוערך בעזרת תנועה מדודה (אינטגרציית מהירות על מקטע קצר וישר, שבו הסחיפה זניחה).
  2. מדידת הגודל הזוויתי של השער בפיקסלים במספר מרחקים.
  3. פתרון ה-FOV שממזער את שגיאת העקביות בין המרחקים; בדיקת reprojection על ההקלטות.
  4. עדכון הערך ב-`ParamSet`; ההקלטות נשמרות כ-fixtures לרגרסיה.
- ‏sanity חוצה-חיישנים: מרחק-מ-PnP לאורך גישה ישרה מול אינטגרציית המהירות האינרציאלית — סטייה שיטתית מסגירה שגיאת FOV או שגיאת מידות שער.

## 5. גיאומטריית החלון העיוור (commit)

הבעיה המובנית של FPV: ככל שמתקרבים לשער, הוא גדל בתמונה — ובנקודה מסוימת גולש מחוץ ל-FOV. מהרגע הזה ועד המעבר אנחנו עיוורים לשער.

```
                    │◄────── d_blind ──────►│
   ────────────────────────────────────────╫──────
   approach (עם ראייה)     ●               ║ שער
                     נקודת ה-commit:       ║
                     detection אחרונה      ╫
```

- ‏`d_blind` — המרחק שבו השער יוצא מה-FOV — נגזר ישירות מהגיאומטריה: חצי-רוחב שער חלקי טנגנס של חצי ה-FOV (בקירוב, לגישה צירית). עם FOV רחב טיפוסי זה סדר גודל של מטרים בודדים.
- **התנהגות commit** (‏`race_planner.py`): בכניסה לחלון — נעילת וקטור המהירות דרך מרכז השער, לפי ה-gate_rel האחרונה האמינה; טיסה על הווקטור הנעול למשך `commit_duration_s`; ‏dead-reckoning ממשיך לעדכן את ה-pose אך לא משנה את הפקודה.
- **אישור מעבר**: אך ורק אינקרמנט של `active_gate_index` (האורקל היחיד). התקבל — מעבר ל-search/approach של השער הבא. לא התקבל בתוך החלון + מרווח — ‏recover: בלימה ל-hover וחיפוש מחדש.
- הפרמטרים `commit_distance` (מתי נועלים) ו-`commit_duration_s` הם מהקריטיים בטיחותית — נעילה מוקדמת מדי מגדילה את השגיאה הצידית במעבר; מאוחרת מדי נשענת על detection בזווית רעה. שניהם ברשימת הכוונון הראשונה של הקמפיינים (מסמך [04](04-lulaat-lemida.md)), ותקציב הסחיפה שלהם מנותח במסמך [02](02-bakara-veshichzur-matzav.md).

## 6. סבב 2 והממשק הפלגבילי — `interface.py`

### 6.1 החוזה

```python
class GateDetector(ABC):
    @abstractmethod
    def detect(self, frame_bgr: np.ndarray, t_sim: float) -> Detection | None:
        """מחזיר פינות שער מסודרות + מדדי איכות, או None."""
```

כל מה שאחרי הפינות — ‏PnP, סינון, פרסום — משותף ואינו תלוי בגלאי. זו נקודת ההחלפה היחידה לסבב 2.

### 6.2 אסטרטגיית סבב 2

סביבות 3D-scanned שוברות את הנחת "שער בניגודיות גבוהה על רקע נקי". התוכנית:

1. **דאטה קודם**: הקלטות מסביבות סבב 2 (דרך `simtools/record.py`) הן תנאי מקדים; הן הופכות ל-fixtures שהגלאי החדש נמדד עליהם offline — בלי לגעת בסימולטור.
2. **מדרגת ביניים קלאסית**: התאמת ה-HSV (ספים אחרים, אולי edge cues) — נמדדת על ה-fixtures; אם מספיקה, נשארים קלאסיים.
3. **גלאי נלמד**: מודל keypoint/segmentation קל שמחזיר את אותן ארבע פינות מאחורי אותו ABC. ה-PnP וכל ה-downstream לא משתנים. תיוג: פריימים מהקלטות סבב 1 שבהם הגלאי הקלאסי בטוח מאוד משמשים כתיוג אוטומטי ראשוני; סבב 2 מתויג ידנית במידת הצורך.
4. **תקציב זמן עיבוד**: הגלאי רץ ב-Perception thread ולא בלולאה החמה, אבל קצב detection נמוך מדי מרחיב את חלונות ה-dead-reckoning — היעד נשאר עיבוד בקצב המצלמה.

קריטריון הקבלה של Phase 6: הגלאי החדש עובד על הקלטות סבב 2, מאחורי אותו ממשק, עם אותם מדדי כיסוי שנמדדו בסבב 1 (‏detection ביותר מ-80% מהפריימים בטווח 15m).

## 7. מדדי איכות ראייה (נרשמים בכל טיסה)

| מדד | הגדרה | שימוש |
|---|---|---|
| ‏detection rate | אחוז פריימים עם detection תקפה בטווח 15m | קריטריון קבלה Phase 3 |
| ‏reprojection error | ממוצע פר detection | בריאות כיול FOV / מידות שער |
| ‏gate_rel_age_s | גיל העדכון החזותי בכל tick | חשיפת חורי ראייה; מזין watchdog |
| ‏frame loss | פריימים שנזרקו ב-GC של `vision_rx` | בריאות רשת/סימולטור |
| ‏pipeline latency | ‏sim_time_ns של הפריים מול זמן פרסום ה-Detection | תקציב ההשהיה של לולאת הראייה |

כל המדדים נכתבים ללוג ה-JSONL ומסוכמים ב-plots הפוסט-טיסה של `telemetry/` — כך רגרסיה בראייה נראית מיד, עוד לפני שהיא הופכת להתרסקות.
